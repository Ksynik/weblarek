```
Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.
Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

interface IProduct {
  id: string; // Уникальный идентификатор товара
  description: string; // Описание товара
  image: string; // Ссылка на изображение товара
  title: string; // Название товара
  category: string; // Категория товара
  price: number | null; // Цена товара, может быть null
}
```
**Назначение:** Описывает структуру объекта товара, который отображается в каталоге и может быть добавлен в корзину.

## Модели данных

Для работы с данными в приложении реализованы три класса-модели, каждый из которых отвечает за свою область данных.

### Класс ProductCatalog

**Назначение:** Хранит массив всех товаров, доступных для покупки, и товар, выбранный для подробного отображения.

**Конструктор:**
```typescript
constructor()
```
Конструктор не принимает параметры.

**Поля класса:**
- `products: IProduct[]` — массив всех товаров, доступных в каталоге.
- `selectedProduct: IProduct | null` — товар, выбранный для подробного просмотра.

**Методы класса:**
- `setProducts(products: IProduct[]): void` — сохраняет массив товаров, полученный в параметрах.
- `getProducts(): IProduct[]` — возвращает массив товаров из модели.
- `getProductById(id: string): IProduct | undefined` — возвращает товар по его идентификатору.
- `setSelectedProduct(product: IProduct): void` — сохраняет товар для подробного отображения.
- `getSelectedProduct(): IProduct | null` — возвращает товар для подробного отображения.

### Класс Cart

**Назначение:** Хранит массив товаров, выбранных пользователем для покупки.

**Конструктор:**
```typescript
constructor()
```
Конструктор не принимает параметры.

**Поля класса:**
- `items: IProduct[]` — массив товаров, добавленных в корзину.

**Методы класса:**
- `getItems(): IProduct[]` — возвращает массив товаров в корзине.
- `addItem(product: IProduct): void` — добавляет товар в корзину.
- `removeItem(product: IProduct): void` — удаляет товар из корзины.
- `clear(): void` — очищает корзину.
- `getTotalPrice(): number` — возвращает стоимость всех товаров в корзине.
- `getCount(): number` — возвращает количество товаров в корзине.
- `hasItem(id: string): boolean` — проверяет наличие товара в корзине по его идентификатору.

### Класс User

**Назначение:** Хранит данные пользователя, необходимые для оформления заказа.

**Конструктор:**
```typescript
constructor()
```
Конструктор не принимает параметры.

**Поля класса:**
- `payment: TPayment` — способ оплаты.
- `email: string` — email пользователя.

**Методы класса:**

`setEmail(email: string): void` — сохраняет email пользователя.

`getData(): IUser` — возвращает все данные пользователя.
`clear(): void` — очищает данные пользователя.
`validate(): boolean` — выполняет валидацию данных пользователя.

## Слой Представления (View)

Ниже описана архитектура и назначение всех компонентов слоя View. Каждый класс отвечает за свой блок разметки, использует соответствующий шаблон из разметки (`<template id="...">`) и наследуется от базовых классов, где это уместно.

### Базовые классы

- Component<T>
  - Назначение: базовый каркас для всех компонент представления.
  - Публичные методы:
    - render(data?: Partial<T>): HTMLElement — проставляет данные через сеттеры и возвращает корневой элемент.
  - Помощники:
    - setImage(img: HTMLImageElement, src: string, alt?: string) — установка src/alt.

- Form<T> (базовый класс для форм)
  - Назначение: общий функционал для всех форм (валидация, очистка, сабмит).
  - Контракты:
    - onSubmit?: (data: T) => void — обработчик успешной отправки.
    - clear(): void — очистка формы, сброс состояний.
    - Валидация включена в конкретных реализациях и управляет доступностью submit.

### Карточки каталога (общий родитель)

- CardBase
  - Родитель для карточек, инкапсулирует общий функционал.
  - Сеттеры:
    - id: string — идентификатор товара.
    - title: string — заголовок карточки.
    - price: string — строковое отображение цены (для компактной карточки каталога).
  - Использование: не рендерится напрямую, а через наследников.

#### Наследники CardBase (работают с разными темплейтами)

- CatalogCard (template: #card-catalog)
  - Назначение: карточка списка товаров на главной.
  - Сеттеры: id, category, image, title, price.
  - События: при клике по всей карточке эмитит `card:select` с `{ id }`.

- PreviewCard (template: #card-preview)
  - Назначение: предпросмотр товара в модальном окне.
  - Сеттеры: image, alt, category, categoryClass, title, description, price (number | null), buttonText, buttonDisabled.
  - Колбэки: onClick — действие по нажатию на кнопку (купить/удалить).

- BasketItem (template: #card-basket)
  - Назначение: элемент списка корзины.
  - Сеттеры: index, title, price (number | null), onDelete — удаление позиции.

### Контейнеры

- Gallery
  - Назначение: контейнер каталога на главной странице.
  - Сеттеры: catalog: HTMLElement[] — массив отрендерённых карточек.
  - Разметка: `<main class="gallery"></main>`.

- Basket (template: #basket)
  - Назначение: состав корзины, итог и кнопка оформления.
  - Сеттеры: items: HTMLElement[], total: number.
  - Поведение: при пустом списке показывает «Корзина пуста» и дизейблит кнопку.
  - Колбэки: onOrder — переход к оформлению.

- Header
  - Назначение: шапка с иконкой корзины и счётчиком.
  - Сеттеры: counter: number — количество товаров.
  - Колбэки: onBasketClick — открытие корзины.

- ModalView
  - Назначение: универсальный контейнер модального окна.
  - API: content (HTMLElement | null), open(), close().
  - События (через публичный `events`): `modal:open`, `modal:close` — для блокировки скролла страницы.
  - Примечание: модалка не имеет наследников; в ней отображаются самостоятельные компоненты (PreviewCard, Basket, формы, Success и т.д.).

### Формы (общий родитель — Form)

- OrderForm (template: #order)
  - Поля: способ оплаты (TPayment), адрес доставки.
  - Сеттеры: payment, address; onPaymentChange(cb).
  - Валидация: «Далее» активно на валидных значениях (выбран метод и непустой адрес).

- ContactsForm (template: #contacts)
  - Поля: email, phone.
  - Сеттеры: email, phone.
  - Валидация: «Оплатить» активно при корректных данных.
  - Сообщения ошибок: «Укажите email», «Неверный формат email», «Укажите телефон», «Неверный формат телефона».

### Завершение заказа

- Success (template: #success)
  - Назначение: экран успеха после оформления.
  - Сеттеры: totalPrice: number; onClose(cb) — закрыть, очистить состояния.

### Поток событий (связь Presenter ↔ View)

- CatalogCard → `card:select` — презентер открывает PreviewCard в модалке.
- PreviewCard.onClick — добавление/удаление из корзины, обновление счётчика, закрытие модалки.
- Header.onBasketClick — открытие корзины в модалке.
- Basket.onOrder — открытие формы заказа (OrderForm) в модалке.
- OrderForm: onSubmit — переход к ContactsForm; onPaymentChange — синхронизация способа оплаты.
- ContactsForm: onSubmit — формирование и отправка заказа; при успехе — Success.
- ModalView: `modal:open`/`modal:close` — блокировка/разблокировка скролла body.

### Тестирование компонентов View

Для быстрой проверки можно использовать контейнер галереи:

1. Найдите в разметке `<main class="gallery"></main>` и сохраните в константу через `ensureElement`.
2. Создайте экземпляр нужного компонента из соответствующего `<template>` (например, `#card-catalog`).
3. Вызовите `gallery.replaceChildren(component.render())` для проверки отображения и поведения.

Таким образом, каждый компонент изолированно отвечает за свой участок DOM, общая логика вынесена в родительские классы, а модальное окно остаётся универсальной оболочкой без наследников.

## События приложения

В этом разделе собраны события и колбэки, которые генерируются слоями View/Model. При добавлении новых событий дополняйте список.

### Модельные события

- ProductCatalog
  - `products:changed` — данные каталога обновлены. Параметры: `IProduct[]`.

### События и колбэки компонентов View

- CatalogCard
  - Событие: `card:select` — пользователь выбрал карточку для просмотра. Параметры: `{ id: string }`.

- PreviewCard
  - Колбэк: `onClick()` — пользователь нажал кнопку действия (купить/удалить). Решение, что делать, принимает презентер.

- Header
  - Колбэк: `onBasketClick()` — пользователь открыл корзину из шапки.

- Basket
  - Колбэк: `onOrder()` — пользователь нажал кнопку «Оформить».

- BasketItem
  - Колбэк: `onDelete()` — пользователь удалил позицию из корзины.

- OrderForm
  - Колбэк: `onPaymentChange(method: TPayment)` — смена способа оплаты.
  - Колбэк: `onSubmit(data)` — переход ко второй форме («Далее»), вызывается при валидных данных.

- ContactsForm
  - Колбэк: `onSubmit(data)` — отправка контактных данных («Оплатить»), вызывается при валидных данных.

- ModalView
  - События: `modal:open`, `modal:close` — изменение состояния модального окна (используются для блокировки скролла и побочных эффектов).

- Success
  - Колбэк: `onClose()` — пользователь закрыл экран успешного оформления.

Примечания:
- В компонентах View предпочтение отдаётся колбэкам (передаются в конструктор/через сеттеры). Глобальные события брокера применяются для межкомпонентных сигналов (например, выбор карточки/состояние модалки).
- Компоненты View не принимают решений о бизнес‑логике: они лишь сообщают о действиях пользователя (через события/колбэки). Решения и изменения данных выполняет презентер/модели.

## Презентер

В приложении используется один презентер, так как страница одна. Презентер реализован в основном скрипте `src/main.ts` без выделения отдельного класса. Такой подход выбран для прозрачности потока событий и снижения накладных расходов на инъекцию зависимостей, при этом соблюдены принципы MVP:

- Презентер подписывается на события Моделей (например, `products:changed`, `cart:changed`, `user:changed`) и обновляет Представления.
- Презентер подписывается на события Представлений/колбэки (например, `card:select`, `onBasketClick`, `onOrder`, `onSubmit`) и вызывает методы моделей/сервисов.
- Презентер не хранит данные и не генерирует собственные события «ради себя»; он только маршрутизирует существующие события и вызывает необходимые действия.
- Перерисовка Предствалений выполняется только:
  - по событию от Моделей (данные изменились);
  - при открытии модального окна (смена контента модалки).

Основные обязанности презентера:
- Инициализация сервисов, моделей и представлений; привязка шаблонов.
- Загрузка каталога (`ServerService.getProductList`) → сохранение в `ProductCatalog.setProducts`.
- Рендер каталога по `products:changed` через `Gallery` и `CatalogCard`.
- Открытие предпросмотра по `card:select`; управление состоянием кнопки «В корзину/Удалить/Недоступно»; добавление/удаление из `Cart`.
- Открытие корзины из `Header`; сбор списка `BasketItem`; удаление позиций; показ итога.
- Оформление заказа: `OrderForm` (шаг 1) → `ContactsForm` (шаг 2) → `ServerService.createOrder` → `Success` и очистка `Cart`/`User`.
- Блокировка/разблокировка прокрутки страницы по событиям модалки `modal:open`/`modal:close`.

Если потребуется масштабирование, код презентера легко вынести в отдельный класс (например, `AppPresenter`) с явной передачей зависимостей (сервисы, модели, представления) через конструктор. Это сохранит слабую связанность и тестопригодность, не нарушая MVP.
